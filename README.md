# Personalized AI Agent Framework

This framework implements a cooperative system of AI agents that can coordinate to complete tasks in automation/workflows, research/information gathering, data processing/analysis, and custom use cases.

## Features

- **Agent-based architecture**: Specialized agents for research, data processing, workflow orchestration, file operations, communication, and monitoring.
- **Message passing**: Agents communicate via a publish-subscribe message bus and point-to-point queues.
- **Shared state**: A blackboard (JSON file) provides shared state accessible by all agents.
- **Hierarchical supervision**: A supervisor agent assigns tasks, monitors progress, and can intervene.
- **Supervised autonomy**: Agents operate independently but report progress to the supervisor.
- **Continuous service**: Monitors a directory for JSON task files and processes them in the background.
- **Real capabilities**:
  - Web scraping using DuckDuckGo (research agent)
  - Data processing and statistical analysis (data agent)
  - Automated Markdown report generation (workflow agent)
  - File operations (file agent)
  - Simulated HTTP/email communication (communication agent)
  - System monitoring (monitoring agent)

## Components

- `agent.py`: Base agent class with lifecycle management and internal task queue
- `message_bus.py`: Publish-subscribe messaging and shared blackboard
- `tasks.py`: Task registry defining types and required agent capabilities
- `supervisor.py`: Supervisor agent for task assignment and monitoring
- `utils.py`: Helper functions for logging and JSON operations
- `research_agent.py`: Web search and summarization agent
- `data_agent.py`: Data processing and statistical analysis agent
- `workflow_agent.py`: Workflow execution and report generation agent
- `file_agent.py`: File operations agent (read/write/copy/search/list)
- `communication_agent.py`: Communication agent (HTTP/email/internal messaging)
- `monitoring_agent.py`: System monitoring agent (CPU, memory, disk)
- `example.py`: Continuous background service demonstrating the framework
- `blackboard.json`: Shared state repository (initialized empty)
- `agent.log`: Consolidated log file for all agents

## Usage

### Running the Continuous Service

The `example.py` script demonstrates a continuous service that:
1. Initializes the supervisor and all agents
2. Assigns initial demonstration tasks (research, data analysis, workflow with report)
3. Monitors the `tasks_input/` directory for JSON task files
4. Processes assigned tasks and moves completed files to `tasks_archive/`
5. Runs until interrupted with Ctrl+C

```bash
python example.py
```

### Task Format

Place JSON files in the `tasks_input/` directory with the following format:

```json
[
  {
    "type": "research",
    "payload": {
      "query": "your search query",
      "max_results": 10
    }
  }
]
```

Supported task types:
- `research`: Web search and summarization
- `data_processing`: File-based data analysis (CSV/Excel)
- `workflow_execution`: Multi-step workflow orchestration
- `file_operations`: Read/write/copy/search/list files
- `communication`: Simulated HTTP/email messaging
- `monitoring`: System metrics collection

The service will process each task file and move it to `tasks_archive/` upon completion.

### Monitoring Progress

- Check `blackboard.json` for current task statuses and results
- View `agent.log` for consolidated logging from all agents
- Reports generated by the workflow agent appear in the `outputs/` directory

## Architecture Overview

### Communication Mechanisms

1. **Topic-based Publish-Subscribe**: 
   - Agents publish to topics like `task_completed`, `agent_registry`
   - Supervisor subscribes to `task_completed` for completion notifications
   - Agents subscribe to `agent_messages_<agent_id>` for incoming tasks

2. **Shared Blackboard**:
   - Persistent JSON file (`blackboard.json`) storing task states and results
   - Agents read/write state via `get_state()`/`publish_state()` methods
   - Supervisor uses blackboard for task tracking

3. **Point-to-Point Queues**:
   - Internal task queues in each agent for asynchronous processing
   - Background worker threads execute tasks without blocking message handling

### Task Lifecycle

1. **Assignment**: Supervisor assigns task via `assign_task()`
   - Validates payload against task registry
   - Selects appropriate agent based on capabilities
   - Publishes task to agent's topic: `agent_messages_<agent_id>`
   - Records task in blackboard with status `assigned`

2. **Processing**:
   - Agent receives task via message handler
   - Task placed in internal queue for background processing
   - Worker thread executes `execute_task()` method
   - Agent publishes result to blackboard and sends completion notification

3. **Completion**:
   - Supervisor receives `task_completed` message
   - Updates task status in blackboard to `completed`
   - Logs completion and optionally notifies requester

## Extending the Framework

### Adding New Agent Types

1. Create a new Python file inheriting from `BaseAgent` (in `agent.py`)
2. Implement `execute_task()` method for your specific functionality
3. Optionally implement `handle_query()` for direct queries
4. Register the agent in `tasks.py` TASK_REGISTRY
5. Initialize the agent in `example.py` or your application

### Adding New Task Types

1. Define the task type in `tasks.py` TASK_REGISTRY
   - Specify required capabilities and example agent
2. Implement validation logic if needed in `validate_task_payload()`
3. Ensure agents advertising the capability can handle the task type
4. Use the new task type in JSON task files or direct assignment

## Design Principles

- **Loose Coupling**: Agents communicate only through the message bus and blackboard
- **Fault Tolerance**: Error handling in agents prevents crashes from affecting others
- **Scalability**: Background threads allow concurrent task processing
- **Transparency**: Comprehensive logging aids debugging and monitoring
- **Extensibility**: Clear interfaces for adding new agents and task types
